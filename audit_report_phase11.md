# Phase 11/12: Coding Mode Audit Report

## 1. Syntax Highlight Integrity
- **The DOM Reconstruction Challenge**: Normal typing engines use a flat string wrapped in `.char` tags. When doing syntax highlighting, standard libraries (Prism/Highlight) chunk words (e.g. `<span class="keyword">function</span>`). 
- **The Solution**: `codingEngine.js` parses the DOM tree generated by Prism, breaks down text nodes char-by-char, and reconstructs the DOM. This ensures that every individual keystroke acts as a collision boundary while seamlessly inheriting its parent syntax coloring.

## 2. Security & Uploads
- **Untrusted Ingestion**: Users can upload `.js` scripts to practice typing. This is a massive XSS vector.
- **Sanitization**: All `content` payloads passing through `upload_practice` are strictly `escaped()`. Furthermore, in `coding.html`, the payload is rendered into a hidden `<textarea>` via `{{ practice.content|safe }}` which treats the escaped text correctly without executing `<script>` tags, before the JavaScript engine reads its `value`.

## 3. Performance & Scalability
- **Smooth Navigation**: `charEl.scrollIntoView({ behavior: "smooth", block: "center" })` automatically ensures that long scripts gracefully scroll down the container, keeping the cursor centered just like a real IDE.
- **Analytics Bridge**: The custom coding test still hooks into `/analytics/api/save-metrics/`. This means all the symbols (`{}()[]=;`) explicitly typed in coding mode are funneled into the AI Analytics engine to update the user's `weak_keys` heatmap.
